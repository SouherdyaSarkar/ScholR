# Scholr
### *Your notes are everywhere. Your mind shouldn't be.*

---

A student has: lecture slides in one folder, handwritten notes in another, syllabus buried in an email, reference book chapters scattered across downloads, and four subjects bleeding into each other the week before exams. General purpose LLMs can't help with that because they have no map of that specific student's world.

What Scholr actually provides is a **personal academic operating system** — the LLM is the engine, but the product is the organisation layer. That reframe also helps with positioning: you're not competing with ChatGPT, you're competing with the student's desktop folder structure and sticky notes.

---

## Core Philosophy

- **Organise first, answer second.** Uploaded material defines the student's academic map — not the source of truth for responses.
- **LLM as engine, structure as product.** The AI's parametric knowledge handles explanations, analogies, and problem-solving. The uploaded context handles syllabus structure, covered modules, professor-specific notes, and exam patterns.
- **Frictionless ingestion.** If throwing files in feels like work, students won't do it. The system does the heavy lifting — parsing, sorting, tagging — so the organised output feels like magic.
- **Isolated memory per subject.** OS knowledge never bleeds into a CN query. Every subject lives in its own context, the way a student's mind (ideally) works.

## Features

### 1. Student Dashboard
A minimal, glanceable home screen containing only what matters:
- Today's study goal and time logged
- Active subjects with last-studied timestamps
- Weak topic flags surfaced from quiz and flashcard performance
- Pending review cards (spaced repetition queue)
- Recent file uploads and their processing status

The dashboard is read-only and contextual — it reflects the student's actual activity, not a generic checklist.

### 2. Workspace: Semester & Subject Hierarchy

```
Semester 1
├── Operating Systems
│   ├── Syllabus.pdf
│   ├── Lecture 3 - Scheduling.pptx
│   └── My Notes - Week 4.pdf
├── Computer Networks
│   ├── Forouzan Chapters 1-5.pdf
│   └── CN Lab Manual.pdf
└── [Root-level files: Timetable, Exam Schedule]
```

- **Root containers** (Semesters) hold global, cross-subject files — timetable, exam schedule, university notices.
- **Subject containers** hold subject-specific material only.
- Root-level files feed a **semester-scoped shared context**; subject-level files feed that **subject's isolated vector DB only**.
- Subjects can be created manually or auto-generated by parsing an uploaded syllabus.

### 3. Per-Subject Memory Base

Each subject maintains its own persistent memory layer:

- Dedicated vector database storing embeddings of all uploaded material
- Tracks which modules/chapters have been covered based on uploaded content
- Indexes professor notes, annotations, and handwritten scans separately from textbook content
- Versioned — new uploads extend the memory base; old context is not overwritten

This memory base is used for **framing** responses, not for generating them. The LLM is told what the student has covered and what their material says — then answers from its own knowledge, anchored to that map.

### 4. File Ingestion Pipeline

Accepted formats: PDF, PPTX, DOCX, images (scanned notes via OCR), plain text.

Ingestion flow per uploaded file:
1. **Parse** — extract raw text, OCR images, detect structure (headings, bullet points, tables)
2. **Classify** — identify if content is a syllabus, lecture slide, textbook chapter, or personal notes
3. **Chunk** — split into semantically coherent segments (not fixed token windows)
4. **Embed** — generate vector embeddings and store in the subject's vector DB
5. **Index metadata** — store source file name, upload date, detected topic tags, page ranges
6. **Syllabus detection** — if a syllabus is detected, auto-extract module/unit structure and scaffold the subject's topic map

Root-level files follow the same pipeline but write to the semester-scoped shared index.

### 5. Voice-Enabled Chat Panel

A dedicated study interface designed for active learning, not passive lookup:

- **Voice input** via Whisper (or equivalent STT) — ask questions hands-free while reviewing notes
- **Voice output** via TTS for conversational responses — useful while commuting or away from screen
- Context-aware per session — each chat session is scoped to a subject (or optionally cross-subject)
- Response style adapts to query type: explanations are detailed, definition lookups are concise, problem-solving is step-by-step
- Session history is retained within the app for reference but is not fed back into the vector DB

### 6. Syllabus-Aware Study Mode

When a student says *"Let's study OS today"*, the system does not open a blank chat. It responds with awareness of the academic map:

> *"You're studying Operating Systems. Based on your syllabus, you have 6 modules — you've uploaded material for Modules 1, 2, and 4. Module 3 (CPU Scheduling) has no notes yet. Would you like to go through Module 3 today, or dive deeper into something specific?"*

This behaviour is driven by curated system prompts that inject:
- The subject's module/unit structure
- Which modules have uploaded material
- Quiz and flashcard performance data per topic
- Time since last study session per module

### 7. Flashcards

- Auto-generated from uploaded material on ingestion (can be reviewed and edited)
- Manually creatable within any subject
- Spaced repetition scheduling (SM-2 or similar algorithm)
- Performance tracked per card and aggregated per topic
- Weak topic flags fed back to the dashboard and study mode prompts

### 8. Short Exams & Quizzes

- Generated on-demand or scheduled: *"Quiz me on OS Module 2"*
- Question types: MCQ, short answer, fill-in-the-blank
- Answers evaluated by the LLM against correct concepts (not just keyword matching)
- Per-question feedback with explanation
- Results stored and aggregated to identify persistent weak areas
- Can be seeded from uploaded past exam papers if provided

## System Design

### High-Level Architecture

```
┌─────────────────────────────────────────────────────┐
│                  Client (Web / Mobile)              │
│   Dashboard · Workspace · Chat · Flashcards         │
└──────────────────────┬──────────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────────┐
│                  API / Backend                       │
│   Auth · Session Management · Query Router          │
└──────┬────────────────────────────────┬─────────────┘
       │                                │
┌──────▼──────────┐          ┌──────────▼──────────────┐
│  LLM Service    │          │   Ingestion Service      │
│  (Orchestration)│          │   Parse · Chunk · Embed  │
└──────┬──────────┘          └──────────┬───────────────┘
       │                                │
┌──────▼────────────────────────────────▼───────────────┐
│                    Data Layer                          │
│   Vector DBs (per subject) · Relational DB (metadata) │
│   File Storage · User Profiles · Progress Records     │
└───────────────────────────────────────────────────────┘
```

### Key Components

| Component | Responsibility |
|---|---|
| **Query Router** | Determines which subject's vector DB to query, or whether to query across multiple |
| **Ingestion Service** | Parses, chunks, embeds, and classifies uploaded files |
| **Prompt Builder** | Assembles system prompts by injecting syllabus map, performance data, and retrieved context |
| **LLM Orchestrator** | Sends assembled prompts to the underlying model, handles streaming responses |
| **Vector Store (per subject)** | Stores and retrieves subject-specific embeddings |
| **Progress Store** | Tracks flashcard scores, quiz results, study time per module |
| **Scheduler** | Manages spaced repetition intervals and study reminders |

## RAG Pipeline Flow

### Standard Query (Single Subject)

```
User query: "Explain Banker's Algorithm"
     │
     ▼
1. Intent Detection
   └── Subject: Operating Systems
   └── Query type: Concept explanation
     │
     ▼
2. Vector DB Retrieval (OS subject store)
   └── Retrieve top-k chunks related to "Banker's Algorithm"
   └── Retrieved: Lecture 5 slide excerpt + personal note snippet
     │
     ▼
3. Context Framing (NOT source of truth)
   └── "Student's material mentions Banker's Algorithm
        in the context of deadlock avoidance.
        Their notes reference resource allocation tables.
        Module 4 covers this topic per syllabus."
     │
     ▼
4. Prompt Assembly
   └── System prompt: study persona + subject context + performance data
   └── Retrieved chunks: framing layer
   └── User query: the actual question
     │
     ▼
5. LLM Response
   └── Answers from parametric knowledge
   └── Anchored to covered syllabus scope
   └── References student's own material where relevant
     │
     ▼
6. Response Delivery
   └── Streamed to chat panel
   └── Cited source chunks shown optionally ("from your Lecture 5 notes")
```

### Cross-Subject Query

```
User query: "Compare flow control in OS and CN"
     │
     ▼
Query Router detects multi-subject intent
     │
     ├──── Retrieval: OS vector store
     └──── Retrieval: CN vector store
               │
               ▼
        Merged context passed to LLM
        LLM synthesises comparison from both stores + own knowledge
```

## Memory Context Preservation

### What Lives in Memory

| Layer | Scope | Contents |
|---|---|---|
| **Subject vector DB** | Per subject, persistent | Embeddings of all uploaded material |
| **Syllabus map** | Per subject, structured | Module names, topics, coverage status |
| **Performance index** | Per subject, per topic | Flashcard accuracy, quiz scores, weak flags |
| **Session context** | Per chat session, ephemeral | Conversation history within a session |
| **Semester shared index** | Per semester | Root-level files, cross-subject schedule |
| **Student profile** | Global | Name, enrolled subjects, study preferences |

### What Does NOT Live in Memory

- Raw LLM responses are not stored in the vector DB
- Chat history is retained for reference but does not affect embeddings
- Session context resets between sessions (stateless per conversation)

### Context Injection at Query Time

Every query to the LLM is assembled with:

```
[System Prompt]
  You are a study assistant for {student_name}.
  Subject: {subject_name}
  Syllabus coverage: {module_list_with_status}
  Weak topics (based on recent quizzes): {weak_topic_list}
  Study style preference: {preference}

[Retrieved Context — Framing Only]
  The student's uploaded material contains the following
  relevant excerpts. Use these to frame your response
  (understand their scope and terminology), but answer
  from your own knowledge:
  {retrieved_chunks}

[User Query]
  {query}
```

This separation — framing vs. source-of-truth — is enforced at the prompt level, not the retrieval level.

## Ingestion Intelligence

### Syllabus Auto-Parsing

When a syllabus document is uploaded, the ingestion service attempts to:
- Extract unit/module names and their sub-topics
- Detect recommended readings or reference books
- Identify internal exam dates or assignment deadlines
- Scaffold the subject's topic map automatically

The student reviews and confirms the extracted structure before it is committed.

### Content Classification

Each uploaded chunk is tagged with a content type:

| Tag | Description |
|---|---|
| `lecture_slide` | Formal course content, likely authoritative |
| `personal_note` | Student's own writing, potentially incomplete |
| `textbook_excerpt` | Reference material, comprehensive |
| `past_paper` | Exam questions, used to seed quiz generation |
| `syllabus` | Structural map, not used for concept retrieval |

These tags influence retrieval ranking — for explanations, textbook excerpts rank higher; for exam prep, past papers rank higher.

## Prompt Engineering Strategy

Scholr's quality is largely determined by the prompt layer. Key principles:

- **Persona consistency** — the assistant always behaves as a knowledgeable study companion, not a generic chatbot
- **Scope anchoring** — responses stay within the student's academic scope unless explicitly asked to go beyond
- **Socratic option** — for certain query types (problem-solving, derivations), the assistant asks guiding questions rather than giving direct answers
- **Adaptive verbosity** — short factual queries get concise answers; conceptual questions get structured explanations with examples
- **Progress-informed tone** — if the student has struggled with a topic in quizzes, the assistant flags it and offers a re-explanation without being asked

## Future Considerations

- **Collaborative study rooms** — shared workspaces where study groups can pool uploaded material
- **Exam timetable integration** — auto-prioritise subjects based on proximity to exam dates
- **Handwriting recognition** — native support for tablet/stylus note uploads
- **Offline mode** — cached flashcard and quiz sessions without connectivity
- **Professor-mode** — institutions can push verified syllabus structures and reading lists directly into student workspaces
